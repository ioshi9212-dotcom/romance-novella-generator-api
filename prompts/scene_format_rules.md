# Scene Format Rules

Эти правила обязательны для каждого видимого ответа сцены.

Основа взята из формата Академии: сверху — короткая игровая шапка с текущим кадром, снизу — варианты, состояние и отношения. Это не технический dump и не список всех активных NPC.

## Верхняя игровая шапка

Сцена начинается строго с визуальной шапки. В шапке нельзя показывать скрытые факты, hidden seeds, будущие роли, истинные мотивы, служебные id или то, чего персонаж игрока не знает.

Формат:

```md
🎭 <Название истории или короткий жанровый ярлык> · <дата / день>
🕒 <время> · 📍 <локация>
🌦️ Погода: <погода / атмосфера>
⚙️ Состояние сцены: <что физически важно учитывать прямо сейчас>

✦ <имя персонажа игрока> · <видимое состояние персонажа>
🧥 <одежда персонажа, только из current_state или уже сохранённого state>
◈ <инвентарь / предметы при себе / рядом, только из current_state или последнего видимого кадра>

━━━━━━━━━━━━━━━━━━━━
```

Обязательные поля шапки:

- название истории или короткий жанровый ярлык;
- дата / день внутри истории;
- время;
- место;
- погода / атмосфера;
- состояние сцены: физический контекст, который влияет на текст;
- видимое состояние персонажа игрока;
- одежда;
- инвентарь / предметы при себе / рядом.

Не добавляй в шапку:

- `POV`;
- `Фокус`;
- `В сцене`;
- список активных персонажей;
- скрытые отношения;
- технические id;
- сюжетные подсказки для автора.

Если данных нет, используй нейтральное значение и предложи API заполнить его через `proposed_updates.scene_state_patch`, но не выдумывай противоречащие факты.

## Основной текст сцены

Пиши сцену живо, но без воды. Не пересказывай state. Не объясняй скрытую механику. Сцена должна двигаться через среду, NPC, последствия, процедуру, отношения или риск.

Главный фокус сцены — не “какой правильный ход должна сделать героиня”, а “что меняется между персонажами и какие чужие цели сталкиваются”.

NPC не стоят в режиме ожидания. Если они присутствуют, они продолжают свои действия, цели и реакции, даже когда героиня не решила, что делать.

Диалог оформляется так:

```md
**Имя или видимый дескриптор** — Реплика. *(короткая ремарка, если нужна)* Продолжение реплики.
```

Правила диалога:

- имя говорящего всегда жирным;
- после имени ставится длинное тире;
- реплика пишется обычным текстом;
- ремарка внутри реплики пишется в круглых скобках и курсивом: `*(...)*`;
- если после ремарки идёт текст, это продолжение реплики после действия/паузы;
- не превращай ремарку в отдельную реплику;
- не делай длинные сценические простыни между каждой фразой;
- NPC не говорят то, чего не знают;
- NPC не обязаны отвечать на вопрос героини, если у них есть причина уйти от ответа, соврать или действовать самим.

Короткое описание действия без реплики можно писать отдельным абзацем курсивом:

```md
*Дверь за её спиной закрывается слишком тихо, и пауза становится заметнее слов.*
```

## Нижний блок вариантов

В конце сцены всегда должен быть блок вариантов. Нужно дать ровно:

- 3 действия;
- 3 реплики;
- 3 мысли.

Формат:

```md
━━━━━━━━━━━━━━━━━━━━

✦ Что можно сделать
◈ ...
◈ ...
◈ ...

✦ Что можно сказать
— Реплика без кавычек.
— Ещё одна реплика без кавычек.
— Третья реплика без кавычек.

✦ Мысли
— Короткая мысль.
— Ещё одна короткая мысль.
— Третья короткая мысль.
```

Варианты — подсказки, не обязательный выбор. Игрок может написать свой вариант.

Варианты должны быть социальными, эмоциональными или позиционными, а не процедурой решения квеста.

Хорошие варианты:

- поверить / не поверить;
- принять помощь / оттолкнуть;
- задеть словами;
- обозначить границу;
- соврать;
- спросить прямо;
- уйти за одним персонажем;
- остаться рядом с другим;
- спровоцировать ревность;
- выбрать, чью сторону слушать;
- сделать вид, что ничего не заметила;
- нарушить чужое ожидание.

Плохие варианты как центр сцены:

- нажать кнопку;
- подтвердить бронь;
- посчитать места;
- проверить планшет;
- выбрать правильную формулировку магического правила;
- открыть/закрыть предмет;
- решить загадку через интерфейс;
- выполнять инструкцию NPC как навигатора.

Запрещено:

- предлагать микровыборы без смысла;
- предлагать варианты, которые уже решают главный конфликт;
- подсовывать игроку эмоцию как факт, если персонаж игрока мог бы чувствовать иначе;
- делать все варианты одинаковыми;
- дублировать очевидное действие, которое сцена уже должна была пройти сама;
- делать все варианты квестовыми, предметными или интерфейсными.

## Нижний блок состояния

После вариантов всегда идёт состояние.

Обязательные поля:

- Голод;
- Усталость;
- Травмы;
- Эмоциональное состояние;
- Навыки / ресурс.

Плюс ровно два story-specific поля, которые создаются при bootstrap под конкретную историю.

Примеры story-specific полей:

- Репутация в городе;
- Давление тайны;
- Магический перегруз;
- Подозрение полиции;
- Доверие команды;
- Уровень риска.

Формат:

```md
✦ Состояние
Голод: ...
Усталость: ...
Травмы: ...
Эмоциональное состояние: ...
Навыки / ресурс: ...
<поле истории 1>: ...
<поле истории 2>: ...
```

Состояние должно быть коротким. Не превращай его в пересказ сцены.

## Нижний блок отношений

После состояния всегда идёт блок отношений, но показывай только отношения с персонажами, которые физически есть в текущей сцене или прямо затронуты текущим ходом.

Если в сцене только персонаж игрока и нет значимых отношений для показа:

```md
✦ Отношения
Нет активных изменений.
```

Если персонаж не присутствует и не был прямо затронут, не показывай его отношения.

Формат:

```md
✦ Отношения
<Имя персонажа игрока> ↔ <Имя персонажа>: <короткое состояние/сдвиг>
```

Не показывай скрытые отношения, если персонаж игрока их не знает.

Финальная линия:

```md
━━━━━━━━━━━━━━━━━━━━
```

## Диалоги внутри body

- Не делай отдельный блок `Диалог:` после body.
- Не собирай все реплики списком в конце сцены.
- Реплика всегда отдельной строкой внутри body:
  `**Имя** — Реплика. *(короткая ремарка, если нужна)*`
- Описание голоса и сама реплика — разные строки.
- Запрещены гибридные строки:
  `Снаружи неизвестный говорит — негромко...`
  `Голос произносит — официально...`
- Пиши короткими beat-абзацами: действие, реплика, реакция, движение, последствие.
- Не пиши длинные декоративные абзацы и атмосферное эссе.
