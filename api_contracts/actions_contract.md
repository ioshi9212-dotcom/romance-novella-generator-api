# GPT Actions Contract — v9

Custom GPT is the writer. Railway stores memory, validates JSON, builds scene prompts, and applies state updates.

## Required launch flow

1. User says `начнем` or gives too little setup.
2. GPT calls `getStartQuestionnaire`.
3. User answers the questionnaire in one message. A partial answer is valid; missing details are generated by GPT.
4. GPT calls `createSession` with the exact complete answer in `raw_start_text` and `mode: gpt_actions`.
5. Railway returns `bootstrap_prompt` and `session_id`; session status is `bootstrap_pending`.
6. GPT stages one complete character per `saveBootstrapPart`, then stages `story_plan`, `current_state`, and its authored `director_bible` with hidden lore, hooks, conflicts, reveals, character functions, and an event queue.
7. GPT calls `finalizeBootstrapPreview`; Railway preserves that authored director state, fills only missing technical defaults, derives the remaining runtime sections, and validates the assembled draft.
8. If the response is `bootstrap_repair_required`, GPT repairs only the listed sections and retries silently.
9. Railway saves `pending_bootstrap.json`, returns the human-readable preview, and marks the session `bootstrap_review_pending`.
10. GPT reads all preview chunks, shows the complete preview, and waits.
11. If the user asks for edits, GPT patches only the affected staged parts and finalizes again.
12. If the user confirms, GPT calls `confirmBootstrapPreview`.
13. Railway commits state files and session becomes `active`; GPT may then call `processTurn` with `(начать первую сцену)`.

## Hard rules

- Do not call `applyBootstrapResult` during normal Custom GPT launch flow. It is legacy/direct save.
- Do not call `processTurn` before `confirmBootstrapPreview` succeeds.
- Do not show raw bootstrap JSON to the user unless they explicitly ask.
- Show only the preview during setup review.
- After normal turns, show only `scene.rendered_text`.

## State layout after confirm

```txt
characters/<character_id>.json
state/knowledge/<character_id>.json
state/relationship_pairs/<a>__<b>.json
story_plan.json
current_state.json
```

## Staged bootstrap core must include

- characters
- story_plan
- current_state

Railway derives `protagonist`, `relationships`, `knowledge`, `npc_state`, `future_locks`, `continuity`, `scene_history`, and `turns`. `director_bible` is optional for protocol compatibility, but normal GPT setup must author it; Railway only normalizes or fills missing technical parts. Do not send empty calls for derived sections.

## Runtime scene context

`processTurn` sends GPT the current frame plus complete compact cards only for the player, active/nearby characters, and a tiny event-driven pool of possible entrants. A possible entrant has `scene_presence: candidate_not_present_yet`: it is optional, has not witnessed the scene yet, and becomes present only when the written scene gives it a causal entrance. The first scene may receive at most one such candidate and later turns at most two; candidates use only the free capacity below four cards, while already active/nearby characters are never dropped. Hidden candidates are loaded only when a matching planned reveal has `status: available`, and must use the atomic reveal protocol when introduced.

`story_plan` must include `setting_summary`, `main_premise`, `protagonist_start`, `player_goal`, `central_conflict`, `central_question`, `opening_scene_intent`, `act_structure`, `character_arcs`, `relationship_focus`, `open_threads`, `forbidden_drift`, `current_story_position`, `status_slots`.
